# ADR-0009: i18n — every user-facing string through i18next, en/lt key parity

Status: accepted
Date: 2026-06-10 (documented retroactively)

## Context

The app ships in English and Lithuanian. Strings hardcoded in JSX silently skip
translation, and the gaps are invisible until a Lithuanian user hits them.
Review found the settings sections fully translated while session/workspace
views had drifted into hardcoded strings.

## Decision

- i18next + react-i18next; resources in `src/locales/en.json` and
  `src/locales/lt.json`; language persisted in localStorage.
- **Every user-facing string goes through `t()`** — visible text, `title=`
  tooltips, `placeholder=`, button labels, option labels and hints. The only
  exemptions are developer-facing debug output and log lines.
- Keys are `scope.key`, scope = feature (`provider.*`, `session.*`,
  `workspace.*`, `developer.*`). Plurals use i18next suffix conventions
  (`_other`).
- `en.json` and `lt.json` are kept in **key-for-key parity**. Adding a key to one
  file without the other is a defect; English fallback text in `lt.json` is
  acceptable as a stopgap, a missing key is not.
- Option lists with labels/hints (e.g. tool modes Off/Ask/Auto) store i18n keys
  in the constant and translate at render time, so the constants stay
  module-level.

## Consequences

- Translation completeness is greppable (`title="`, `>"` literals in pages/ are
  suspects) and diffable (JSON key parity).
- Minor ceremony for obviously-internal strings; the exemption list is
  deliberately tiny to keep the rule mechanical.
